Text Input Method

  • SDK
  • Text
  • Text Input Method

    • About MyScript Text Input Method

    MyScript Text Input Method can be used for the fast development of interactive applications enabling handwriting recognition. It recognizes in background the first strokes while the user keeps writing more strokes. That background process starts as soon as the first stroke is written.

    Text Input Method is an SDK extension to MyScript Text. As a wrapper layer to the MyScript Text SDK, it is dependent on certain objects of that parent SDK, such as the resource files and the Text recognizer.

    In brief, MyScript Text Input Method supplies:

    Item Description  Location
    API for C, Java, and .NET. Text Input Method API. voim/api/your_programming_language/
    voim.dll The specific library. voim/bin
    Configuration and manifest files Configuration files (Engine.properties, LanguageManager.properties and Recognizer.properties) and manifest files ( *language_COUNTRY*.lang) allow customizing your recognition. Check the dedicated section for more details. voim/conf/
    BackgroundRecognition This code sample shows how to use the Text Input Method API. voim/api/programming_language/examples/

    MyScript Text Input Method has the following features:

    • Threading
      MyScript Text Input Method enables to carry out background handwriting recognition processing in a separate thread.

    • “Real time”
      MyScript Text Input Method allows obtaining dynamic “real-time” recognition results.

    • Writing styles
      MyScript Text Input Method supports isolated character (boxed) and natural cursive handwriting.

    • Supported characters
      MyScript Text Input Method can provide recognition of all characters of the corresponding alphabet: upper case, lower case, accented characters, digits, punctuation marks, and special characters. It supports the languages supplied as MyScript Lingo language packs.

    If not done already, check the documentation of MyScript Text and the code examples of MyScript Text for standard recognition, as a start. This will give you a better idea of the Text Input Method extension functions.

    Application blueprint

    The figure below shows the relationships between the main API objects.

    1. You need to create an engine object, first of all, to set the runtime environment.

    2. Once the engine is created, you can run handwriting recognition through the voimRecognizer object. In order to send pen events to the voimRecognizer, a recognition mode must be selected. Recognition modes are made available through voimLanguage.

    3. Once the recognition process is completed, an event notifies that a recognition voimResult can be retrieved. Data flow and notification callback use are discussed in the following.

    The recognizer is designed to allow easy implementation of multi-threaded recognition. Multi-threaded recognition is necessary for interactive recognition-enabled applications such as input methods. Therefore, all time consuming tasks are performed in a background thread. Below is a diagram that shows a simple example of execution and data flow between the UI thread(s) and the recognizer worker thread.

    In this scenario, we perform recognition using two different UI side classes: A1 which is executed in a UI thread and A2 which is executed in the recognizer worker thread.

    In the diagram, blue arrows represent API calls, green arrows represent notification event calls, and orange arrows represent internal data flow.

    An important part of the data flow in this type of application is the use of notifications, which are made available through the TextInputMethod API.

    For notification callback management, use the EventListener. This entity is capable of handling notification events that are raised by the recognizer’s worker thread. Keep in mind that the UI thread context is not valid once in the notification callback, when handling communication between the two threads.

    Various events are available for callback, some of which are merely informative. For example, some events simply return the parameters currently in use (e.g. the guidelines used), or the success of a certain action. These are described in the reference documentation.

    Environment

    This section discusses the principal API objects, their usage and the functionalities that they manage.

    Engine

    The engine is the object that models the handwriting recognition engine runtime. It manages:

    • the certificate. This must be declared when you use the engine.

    • properties: when creating a new engine, you pass the Engine.properties file from the configuration you have chosen. (You can also override Engine properties on creation, as they are optional).

    • errors.

    All other objects are bound to one, and only one, engine object.

    Language Manager

    The LanguageManager allows you to manipulate the available languages and configurations. When you create this, you can specify the LanguageManager.properties to use. You can optionally override LanguageManager properties.

    This represents a handwriting language and its associated language modes.

    Dictionary

    The dictionary represents a set of words (or expressions) that can be given to the recognizer as a user dictionary.

    Recognizer

    The recognizer carries out the handwriting tasks. The recognizer object manages a number of important tasks. When creating a new recognizer, you pass the Recognizer.properties file from the configuration you have chosen. (You can also override voimRecognizer properties on creation, as they are optional).

    It is mandatory to use setCallBack after you create a recognizer, in order to receive event notifications.

    Result

    The result object represents the results sent by the recognition engine after recognition. This is linked to the type that is specified in the recognition language mode: words or characters. Various candidate objects are proposed, and the count, label and recognition score can be retrieved.

    EventListener

    This entity is capable of handling notification events that are raised by the recognizer object’s worker thread. Keep in mind that the UI thread context is not valid once in the notification callback, when handling communication between the two threads.

    Resources

    MyScript Text Input Method supports the languages supplied as MyScript Lingo language packs. They contain Alphabet Knowledge (AK) resources and Linguistic Knowledge (LK) resources.

    Alphabet Knowledge
    This resource contains the existing symbols for a given language. It is required as for standard Text recognition.

    Linguistic Knowledge
    This resource provides the information about the words, phrases and semantic unit existing in a given language. It is required as for standard Text recognition.

    For more details on the resources, check the dedicated section.

    Result

    The result consists of a list of candidates that are the proposed recognized samples, as described in MyScript Text SDK.

    Customizing the recognition

    User dictionaries

    The type of applications typically created using MyScript Text Input Method will probably make use of a user dictionary. MyScript recognizer makes use of resources that provide linguistic information to assist handwriting recognition. A dictionary also does this, but it is added and expanded dynamically.

    To enable use of a dictionary, use the setUserDictionary function on the recognizer.

    You add the dictionary contents “on the fly”. This could be used in applications where for example the user can introduce a new word into a personal dictionary at any time. This is relatively convenient, but adding dynamic content may take time.

    To add a word, use the addWordToDictionary function, specifying the dictionary in which to add the word.

    Building a dictionary dynamically can be costly in terms of time. If you want to add linguistic information whenever a certain mode is used, and you do not need to add that information “on the fly”, then you should instead add a resource to the recognition mode concerned, directly in the configuration file. However you will not be able to add content dynamically in the end user application.
    Information on creating custom resources is available in MyScript Text documentation.

    Configuration and manifest files

    MyScript Text Input Method makes many aspects of customizing recognition easier by the use of configurations and manifests. Configurations are defined in .properties files (these are located in the conf directory). Please refer to the comments in the Engine.properties, LanguageManager.properties and Recognizer.properties files for details on how to customize them.

    For each supported language, a set of predefined modes has been defined in an associated manifest. This is expressed as a lg-CT.lang file (for example, en-GB.lang describes the properties concerning the en-GB language and its language modes). The voimLanguageManager handles all actions associated with this during processing (retrieval of languages, language count, mode count, result type etc.).

    Language manifest

    In a language manifest file, a handwriting language and its associated languages modes are defined as follows.

    At the top of the file, there are generic attributes that apply to all the modes it contains:

    Attribute name Description Format Example
    Language-Version The current version x.x 1.0
    Language-Name The language expressed as a language and country code lg-CT en-GB
    Configuration-Script This is a separate section that contains various properties, similar to the scripts used in InkTool for testing. For example: setting the speed-quality compromise, specifying the resource locations
    (See below for details)
    These will apply to all the modes listed.    		 Property value Configuration-Script:
    SetSpeedQualityCompromise 68
    AddResDir ../../../resources

    Then, there are specific attributes for each mode:

    Attribute name Description Value Example
    Name The name of the mode: all the modes in the manifest apply to that particular language. A text string Name: digits
    Input-Type SINGLE_CHARACTER for isolated recognition or SINGLE_LINE_TEXT for a single line of cursive recognition or MULTIPLE_LINE_TEXT for multiple lines of cursive recognition SINGLE_CHARACTER,SINGLE_LINE_TEXT or MULTIPLE_LINE_TEXT Input-Type: MULTIPLE_LINE_TEXT
    Result-Item-Type Describes what form the results will be, single characters or words WORD or CHARACTER Result-Item-Type: WORD
    Use-User-Dictionary specifies whether this mode can use a user dictionary or not true or false Use-User-Dictionary: true
    Configuration-Script This is a separate section, and contains various properties, similar to the scripts used in InkTool for testing. These will apply only to this mode. The available parameters are described below, for the sake of readability. AddResDir ../../../resources
    AddResource en-GB/en_GB-ak-cur.res
    SetSpeedQualityCompromise 68
    SetCharListSize 10
    SetWordListSize 5
    SetTextListSize 6
    SetCharacters 1234567890+-x/.

    All available parameters for Configuration-Script are:

    Parameter Description
    AddResDir Indicates the general resource location
    AddResource Points to a specific resource
    SetSpeedQualityCompromise Sets the recognition speed-quality compromise (0 meaning faster recognition process, 255 meaning better quality)
    SetCharListSize Sets the size of the candidate lists given at character level in the recognition result (between 1 and 20)
    SetWordListSize Sets the size of the candidate lists given at word level in the recognition result (between 1 and 20)
    SetTextListSize Sets the size of the candidate lists given at text level in the recognition result (between 1 and 20)
    SetCharacters Gives the characters that are enabled for recognition: all others will be ignored. Just list the characters you wish to use.
    DiscardAccentuationVariations Turning off accentuation variations avoids redundancy in the top list entries. If you discard the accentuation variations, entries that are identical but with accentuation differences will be ignored. This is a true/false Boolean.
    DiscardCaseVariations When active, this eliminates any results from the top lists that are identical apart from case differences. If case variations are unimportant for your results, this can provide more effective scoring. This is a true/false Boolean.
    SetOverrideAccentuationScheme Ignores all the accentuation schemes specified for all the resources attached to that recognizer. This is an easy way of ignoring the accentuation set for a number of resources without having to modify each of them. This is an integer value: 1 = strict and 2 = lax.
    SetOverrideCasingScheme Ignores all the casing schemes specified for all the resources attached to that recognizer. This is an easy way of ignoring the casing set for a number of resources without having to modify each of them. This is an integer value: 1 = strict, 2 = legacy and 3 = lax.
    EnablePredictor Enables the production of word completion and prediction candidates in the recognition result.
    SetWordCompletionListSize Set this value to greater than 0 so that the engine can return completion candidates (set to zero to turn off this feature). This corresponds to the maximum number of candidates for completion that will be suggested.
    SetWordPredictionListSize Set this value to greater than 0 so that the engine can return prediction candidates (set to zero to turn off this feature). This corresponds to the maximum number of candidates for prediction that will be suggested.

    Here is an example:

    Name: digit
Input-Type: SINGLE_CHARACTER
Result-Item-Type: CHARACTER
Use-User-Dictionary: false
Configuration-Script:
  AddResource en_US/en_US-ak-iso.lite.res
  SetCharacters 0123456789

    This mode will recognize only digits, for US English, in an isolated character (boxed) style, with no user dictionary.

    In the supplied language manifests, the following predefined recognition modes are available:

    • For single character recognition:

      • char

      • digit

    • For multiple character recognition:

      • text

      • email

      • number

      • phone_number

      • uri

    For detail on these defined modes, please refer to the language manifest files (lg-CT.lang) and to the data formats described in the MyScript Text general documentation.

    MUL resources

    MyScript Text Input Method is delivered with special resources, which can recognize certain data formats in all supported languages. These resources have an “mul” prefix and are named in this format: mul-lk-resource_name.res. These resources are located in the resources/mul folder in the MyScript Text installation root folder. You can use these resources in your custom recognition modes (see below).

    Customizing a manifest

    Language manifest files are editable, so you can define your own modes if you wish. In terms of resources to assist recognition, you can use any of the Alphabet Knowledge (AK), Linguistic Knowledge (LK) or Subset Knowledge (SK) resources delivered with your valid MyScript Lingo language packs.

    To do this, simply copy and paste one of the existing modes, then modify the parameters.

    For example, add or remove resources, change or add a dictionary, set the characters that will be enabled etc. Give your mode a unique name so you distinguish it from the default modes.

    Here is an example:

      ORIGINAL MODE
	Name: text
	Input-Type: MULTIPLE_LINE_TEXT
	Result-Item-Type: WORD
	Use-User-Dictionary: true
	Configuration-Script:
	AddResource en_GB/en_GB-ak-cur.lite.res
	AddResource en_GB/en_GB-lk-text.lite.res

	CUSTOM MODE
	Name: diagnosis
	Input-Type: SINGLE_LINE_TEXT
	Result-Item-Type: WORD
	Use-User-Dictionary: false
	Configuration-Script:
	AddResource en_GB/en_GB-ak-cur.lite.res
	AddResource en_GB/en_GB-lk-text.lite.res
	AddResource custom/en_GB/en_GB-lk-medical_terms.res

    This mode will specialize in recognizing medical terminology, using a special custom resource of medical terms (that you would create yourself), on a single line, for United Kingdom English, in a cursive style, with no user dictionary.
